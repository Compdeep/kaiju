Optimistic DAG Execution with Scoped Intent-Gated Tool Control for Autonomous LLM Agents

Cormac Guerin

Autonomous LLM agents that call external tools face two compounding problems: sequential execution that scales quadratically in token cost, and tool-safety enforcement that relies on prompt instructions the model can ignore. We present a system that addresses both by separating the execution layer from the conversational model. The LLM produces a plan; a structurally independent execution layer schedules, gates, and adapts it—the model never observes gate rejections, preventing adversarial iteration against the safety policy. A DAG planner emits a parallel dependency graph of tool calls in a single LLM call. The graph executes optimistically (execute first, adapt on failure) across three adaptive modes—Reflect, nReflect, and Orchestrator—that differ in when and how the system evaluates intermediate results. A four-variable execution gate checks every tool call against structurally assigned scope, caller intent, tool-declared impact, and delegated external clearance, independent of LLM output. On observed benchmarks (99 task executions, 30 per mode), the system completes multi-tool tasks in 7–15 seconds and 3–4 LLM calls versus an estimated 28–90 seconds and 7–18 calls for sequential ReAct, with consistent output quality across all three modes.

1. Introduction

The problem

LLM-based agents that invoke external tools—APIs, shell commands, databases, web services—have become a dominant paradigm for grounding language models in real-world action. The prevailing execution model, ReAct (Yao et al., 2023), operates as a sequential think-act-observe loop: the model reasons about which tool to call, executes it, observes the result, and repeats. This is simple and effective for short tasks. It breaks down on longer ones.

Why is this hard? Two problems compound. First, each ReAct turn carries the full conversation history. Turn i transmits c + ik tokens, where c is the system context and k is the average tool result size. Total cost across n turns is O(n²k)—quadratic in both tools and result size. A 7-tool task costs ~63K tokens; an 18-tool task costs ~250K. Latency scales linearly: each tool blocks on a full LLM round-trip. Second, tool safety is enforced via prompt instructions (“do not call destructive tools”), which the model can ignore through hallucination, prompt injection, or context overflow. There is no structural failsafe.

Why haven’t prior systems solved this? LLM Compiler (Kim et al., 2024) introduced parallel DAG execution but evaluates results only after the entire graph completes, with no mid-execution adaptation and no safety gating. LangGraph models workflows as state machines but keeps the LLM in the execution loop at every routing decision, with safety limited to human-in-the-loop breakpoints. Multi-agent frameworks (CrewAI, AgentArch) add coordination overhead that degrades both performance and reliability—AgentArch found that even the best configurations achieved only 35.3% success on complex enterprise tasks.

Our approach. We present a system that addresses both problems by separating the execution layer from the conversational model. The LLM produces a plan; a structurally independent execution layer schedules, gates, and adapts it. This separation is load-bearing: because the LLM never observes gate rejections during planning, it cannot iterate against the safety policy—a fundamental limitation of conversational agent loops where tool failures feed back into the model’s context. The planner emits a dependency graph of tool calls in a single LLM call. The execution layer processes the graph optimistically (in the concurrency-control sense: execute first, adapt on failure via reflection checkpoints) across three adaptive modes. A four-variable execution gate—composing scope, intent, impact, and delegated clearance—checks every tool call against a structurally derived permission ceiling, independent of LLM output.

Summary of contributions

  1. Optimistic DAG execution with three adaptive modes. A planner emits a parallel dependency graph; three modes differ in how they evaluate intermediate results: structural phase boundaries (Reflect), periodic batch checkpoints (nReflect), and per-node observers (Orchestrator). All three reduce token complexity from O(n²k) to O(nk) or O(nkd) and serial latency from O(n) to O(d).
  2. Parameter dependency injection. A param_refs mechanism enables the planner to express data flow between steps at plan time, with concrete values resolved at execution time from upstream JSON outputs. This bridges the gap between upfront planning and runtime data availability without collapsing to sequential execution.
  3. Iterative reflection with replan loops. When all planned work completes, the system evaluates whether the gathered evidence is sufficient. If not, it replans with new tool calls, executes them, and reflects again—looping until the reflection concludes or budget is exhausted. This ensures multi-phase tasks complete fully rather than stopping after the first wave.
  4. Four-variable scoped intent gate. Every tool call is checked against scope (which tools are visible), intent (how destructive the caller permits), impact (what the tool declares), and clearance (what an external authority approves). The gate is structural—compiled into the tool interface, not communicated via prompt.
  5. Delegated resource clearance. Resource-level authorization is delegated to external HTTP endpoints. The gate sends tool name, parameters, and caller identity; the endpoint returns allow/deny. This enables deployment across hospitals, drones, enterprise IT, and filesystem environments without domain-specific logic in the agent.
  6. Execution layer abstraction. The LLM has no knowledge of parallel scheduling, dependency resolution, parameter injection, or failure recovery. Each LLM call operates on a clean, bounded context. The execution layer translates high-level intent into precise operational actions.

Section 5 presents the execution gate. Section 6 reports experimental results. Section 7 discusses related work. Section 8 concludes.

2. Running Example

A multi-phase system audit

Consider the query: “Check disk usage, list open ports, find git repos under /home, search for CVEs affecting our kernel version, and check environment variables for leaked secrets.” This requires five independent data-gathering branches, a dependency chain (kernel version must be discovered before searching for CVEs), and cross-domain synthesis.

ReAct baseline — 12 sequential turns, cumulative context

turn 1: sysinfo → kernel = 5.10.0-37 (context: 2K) turn 2: disk_usage → / is 78% full (context: 4K) turn 3: net_info → 8 listening ports (context: 7K) turn 4: bash find .git → 25 repos found (context: 11K) turn 5: git status → repo 1 status (context: 13K) ... turn 10: env_list → no leaked keys (context: 28K) turn 11: web_search CVE → 3 advisories (context: 31K) turn 12: synthesize → final report (context: 34K)

Each turn carries the full history. By turn 12, the context exceeds 30K tokens. Early details (the disk usage numbers, the specific kernel version) risk being deprioritized by the model’s attention mechanism. Total cost: ~130K tokens across 12 LLM calls. Latency: 12 serial round-trips.

Our system — parallel DAG, 3 LLM calls

planner (1 LLM call) │ ├── sysinfo ─┐ ├── disk_usage ─┤ ├── net_info ─┤ ← all 7 fire in parallel ├── bash find ─┤ ├── process_list ─┤ ├── env_list ─┤ └── bash uname ─┘ ▼ ◆ reflect (1 LLM call — evaluates all 7 results) │ decision: "replan — need CVE search + git status" │ ├── web_search (query injected via param_refs ← sysinfo.os) ├── git status (path from bash find results) └── git status (second repo) ▼ ◆ reflect (evaluates new results — "conclude") │ ▼ aggregator (1 LLM call — synthesizes all evidence)

Seven tools fire in parallel in the first wave. The reflection evaluates all results, identifies gaps (no CVE search, no git status), and replans with targeted follow-ups. The CVE search uses param_refs to inject the kernel version discovered by sysinfo. A second reflection confirms sufficiency. The aggregator synthesizes everything into a structured report. Total: 3–4 LLM calls, ~22K tokens, 10–15 seconds. The LLM never sees more than one phase of results at a time.

The running example illustrates three key properties. (1) Parallel execution reduces latency from 12 serial round-trips to the depth of the dependency graph (2 phases). (2) The reflection loop ensures multi-phase tasks complete fully—the system does not stop after the first wave of results. (3) Parameter dependency injection bridges plan-time structure and runtime data, enabling the planner to express “search for CVEs affecting whatever kernel version sysinfo returns” without knowing the version at plan time.

3. System Model

Graph structure

The fundamental execution unit is a node—a single discrete operation within a directed acyclic graph. Every node carries a unique identifier, a type, a lifecycle state, upstream dependencies, a parameter map, and a result slot populated upon completion. Nodes are connected by directed edges through depends_on references: a node will not fire until every dependency has reached a terminal state.

The planner LLM produces the initial graph from the task description. During execution, the graph is live—reflection checkpoints, observers, and failure-repair calls may graft additional nodes at any point. Every state transition is emitted as a streaming event, making the full execution trace observable in real time.

Node lifecycle

State transitions: pending ──→ running ──→ resolved (success — result populated) ──→ failed (error — micro-planner may recover) ──→ skipped (cascaded from failed dependency)

A node remains pending until all dependencies reach a terminal state. It then transitions to running, at which point parameter injections are resolved and the operation executes. Terminal states are final—a node is never re-executed. If a node fails, the micro-planner may graft replacement nodes rather than retrying the original.

Edges: ordering and data flow

depends_on establishes execution ordering. A node will not fire until all referenced nodes have completed. Nodes without dependencies execute immediately and in parallel.

param_refs establishes data flow. Each entry maps a parameter name to a source node, a dot-path field selector into that node’s JSON result, and an optional template string. At fire time, the scheduler extracts the value, applies the template if present, and injects the result into the parameter map. Any node referenced in param_refs is automatically added to depends_on.

Example: meeting booking with dependency injection step 0 check_calendars params: {team: "engineering"} step 1 create_event params: {title: "Sync"} param_refs: {start: {step: 0, field: "slots.0.start"}} depends_on: [0] step 2 send_invites param_refs: {event_id: {step: 1, field: "event_id"}, attendees: {step: 0, field: "members"}} depends_on: [0, 1]

Each value materializes exactly when it becomes available. The planner declared the complete three-step workflow upfront without knowing any concrete values—it only specified where each value would come from. The mechanism is structural (the scheduler enforces dependency ordering), fails fast (missing fields produce immediate diagnostic errors), and composable (a single step may reference multiple source nodes).

Node types

Tool Executes a registered tool with declared parameters. Subject to the execution gate. The primary node type.
Reflection LLM checkpoint that evaluates completed results. Can continue, conclude early, or replan by grafting new nodes.
Observer Lightweight per-node LLM call (Orchestrator mode). Evaluates a single result; can inject nodes, cancel pending work, or trigger reflection.
Micro-Planner Scoped failure-repair call. Evaluates a failed node and produces replacement nodes: retry, substitute, or skip.
Interjection Human-triggered checkpoint. The operator’s message gates all pending nodes, giving the ability to redirect or halt an active execution.
Aggregator Final LLM call. Synthesizes all gathered results into a structured output. Fires only after all other nodes complete and the final reflection concludes.

4. Execution Modes

Three modes for three tradeoffs

The planner produces a dependency graph. Nodes without dependencies execute immediately and in parallel. The system then diverges into one of three modes, which differ only in how they evaluate results after each node completes.

Reflect

Reflection checkpoints are injected between dependency waves. At each boundary, the system evaluates all completed results and may replan. Execution pauses only at wave boundaries. Lowest LLM call count.

nReflect

Reflection fires after every N node completions regardless of graph structure. Decouples evaluation frequency from dependency depth. Suited to throughput-sensitive scenarios.

Orchestrator

A lightweight observer evaluates each completed node. Observers can inject follow-ups, cancel pending work, or trigger full reflections. Per-result reactivity at parallel speed. Highest quality output.

Reflect — reflection gates between dependency waves

planner │ ├── tool A ─┐ ├── tool B ─┤ └── tool C ─┤ ▼ ◆ reflect ← evaluates A, B, C; may replan │ ├── tool D ─┐ └── tool E ─┤ ▼ ◆ reflect ← conclude or replan again │ aggregator

Orchestrator — per-node observer with injection capability

planner │ ├── tool A → ○ obs → "continue" ├── tool B → ○ obs → "inject" ──┐ ├── tool C → ○ obs → "cancel E" │ └── tool D → ○ obs → "continue" │ │ tool F (injected) ◄────────────────┘ │ └── ○ obs → "reflect" │ ◆ reflect │ aggregator

Iterative final reflection

A structural guarantee ensures that a reflection evaluates results after every batch of tool completions. When all planned nodes complete, if skill work has occurred since the last reflection, the system injects a new reflection node into the graph. If the reflection replans, the new nodes execute through the normal scheduler loop, and the process repeats. This continues until the reflection concludes, the budget is exhausted, or the wall clock expires. The aggregator fires only after the final reflection approves.

This mechanism is critical for multi-phase tasks. Without it, the system would complete the first wave of tool calls, skip deeper follow-ups, and hand incomplete evidence to the aggregator. With it, the reflection loop drives the task to completion: search → fetch → reflect → need more data → fetch again → reflect → sufficient → aggregate.

Preemption

The graph is preemptable at any point. An operator may inject a message into the active graph. The scheduler creates a gating reflection node with the operator’s message as input. All pending nodes are blocked behind it. Running nodes complete normally, but no new nodes fire until the interjection reflection resolves. The operator’s guidance may continue the plan, conclude early, or replan entirely.

Failure recovery

When a tool node fails, the micro-planner evaluates the error alongside sibling results and produces a replacement subgraph: retry with different parameters, substitute an alternative tool, or skip. Failed nodes are not retried—the replacement is a new node grafted onto the graph with a spawned_by reference to the failed original. This preserves the audit trail.

The execution layer abstraction

The LLM is never told it is part of a pipeline. It receives a task, a set of tools, and evidence gathered so far. It produces a plan, evaluates progress, or synthesizes results. Each LLM call is self-contained: the planner does not know there will be a reflection; the reflection does not know how many parallel steps ran; the aggregator does not know which mode was used. The execution layer surfaces inputs and outputs—tool results, error messages, evidence summaries—while managing scheduling, dependency resolution, and adaptation internally.

The LLM should have no knowledge of its execution layer. Execution mechanics—parallel scheduling, dependency resolution, parameter injection, failure recovery—are infrastructure that do not improve reasoning quality. By abstracting them away, each LLM call operates on a clean, bounded context focused on the task at hand.

5. Execution Gate

Four variables, four sources

Before every tool execution—whether planned, replanned by reflection, or injected by an observer—the gate evaluates four independent concerns. No single entity, including the LLM, controls more than one.

VariableQuestionSet byDescription
ScopeWhich?Policy A composable permission boundary determining which tools the agent may invoke. Defined as allowlists with optional per-tool impact ceilings. Default deny: absent tools are invisible to the planner and rejected by the scheduler. Multiple scopes merge by union (tools) and minimum (caps).
IntentWho?Caller The operational ceiling for the current task. An integer set by the trigger source—never by the LLM. A routine scheduler invokes at intent 0 (observe); a human operator at intent 1 (operate); an explicit override at intent 2.
ImpactWhat?Tool author Declared per-tool at compile time. A read-only query is 0; a write is 1; a deletion is 2. Parameter-aware: the same tool may return different impact depending on its arguments.
ClearanceWhere?External authority Resource-level authorization delegated to an external HTTP endpoint. The gate sends tool name, parameters, and caller identity; the endpoint returns allow/deny. Unreachable or timed-out endpoints result in denial. The agent framework holds no domain-specific authorization logic.
Execution Gate
tool ∈ scope  ∧  tool.impact(params) ≤ min(intent, scope_cap)  ∧  clearance(tool, params, user) = allow
Three checks, evaluated in order of cost. Scope: set membership test. Impact: integer comparison. Clearance: HTTP round-trip (short-circuited when cheaper checks fail). All three are structural—no LLM is involved. The most restrictive ceiling wins.

Three layers of scope enforcement

Layer 1: Planner visibility. The planner receives only tools in scope. It cannot plan what it cannot see.
Layer 2: Scheduler rejection. Even if a tool name appears through hallucination or prompt injection, the scheduler rejects it before the gate.
Layer 3: Impact cap enforcement. A tool in scope may still be capped. A shell tool capped at impact 1 permits diagnostic commands but blocks destructive operations.

Delegated clearance

Resource-level authorization depends on domain knowledge the agent framework cannot possess. A hospital needs patient-ward access rules. A drone needs geofence boundaries. An enterprise needs Active Directory group policies. Rather than encoding domain logic, the gate delegates: it sends a structured request to an external endpoint and respects the boolean response unconditionally.

Clearance protocol: Gate ──→ POST endpoint {tool: "navigate", params: {zone: "zone_3"}, user: "alpha"} ◄── {allow: false, reason: "zone_3 outside authorized airspace"}

The design is domain-agnostic (the framework asks, not reasons), fail-secure (unreachable endpoints deny), and parameter-aware (the same tool can be approved for some invocations and denied for others). For latency-sensitive deployments, the endpoint runs on localhost with sub-millisecond overhead.

Trust model

Two trust boundaries. The tool author declares impact at compile time—the gate enforces the contract. The clearance authority decides resource access—the gate respects the decision. The LLM is trusted with neither. This makes the gate robust to prompt injection: even if adversarial input manipulates the planner into requesting a high-impact tool, the structurally assigned scope, intent, and clearance reject the call.

Intent is never set by the LLM. If the LLM could set its own intent, the gate would be meaningless—equivalent to asking the model to approve its own actions. Intent must originate from outside the LLM’s control surface: a deployment configuration, a process boundary, or an authenticated API caller.

The same pipeline applies identically regardless of how the plan was produced. Whether the planner wrote a JSON array in its text response or called a plan() function via native tool calling, the gate receives the same input: a tool name and a parameter map. The structural separation between LLM output and tool execution is absolute.

6. Experiments

Complexity analysis

Let n = tool calls, d = dependency depth, k = average result size (tokens), c = base context, N = batch size (nReflect). The dominant cost is total tokens transmitted to the LLM.

ReAct: Turn i carries c + ik tokens. Total: ∑(c + ik) ≈ O(n²k). Serial latency: O(n).

Reflect: Planner sees c. Each of d reflections sees accumulated phase results. Aggregator sees all n results. Total: O(nkd). When d « n, approaches O(nk). Latency: O(d).

Orchestrator: Each observer sees exactly one result (k tokens). Total: O(nk)—strictly linear. More LLM calls (n + d + 1) but each individually cheap.

7. Related Work

Positioning

ReAct (Yao et al., 2023) established sequential think-act-observe with O(n²k) token complexity and O(n) latency. AutoGPT and BabyAGI extended this with persistent memory but retained sequential execution.

LangGraph models workflows as state machines with the LLM at every routing decision. Parallelism requires explicit fan-out commands. Safety is limited to human-in-the-loop breakpoints.

LLM Compiler (Kim et al., 2024) is the closest prior work. It generates a DAG, dispatches in parallel, and reports up to 3.7× speedup. Our system extends it in four directions: field-level dependency injection (vs. whole-output substitution), mid-execution adaptation via three modes (vs. post-completion Joiner only), structural safety gating (vs. none), and scoped failure recovery (vs. full replanning).

Multi-agent frameworks (CrewAI, AgentArch) organize agents by role. AgentArch (2025) found that multi-agent ReAct “consistently underperformed” single-agent function calling, with only 35.3% success on complex tasks and 6.3% reproducibility. Our system achieves specialization through the tool interface rather than agent personas.

8. Conclusions

Summary and future work

We presented an optimistic DAG execution layer for autonomous LLM agents that reduces token complexity from O(n²k) to O(nk), serial latency from O(n) to O(d), and enforces tool safety through a four-variable structural gate independent of LLM output. Three execution modes provide a design space trading speed, cost, and depth. Delegated clearance extends the gate to domain-specific resource authorization without domain logic in the agent. On 99 observed task executions, all three modes completed successfully with 3–7× latency improvement over estimated sequential baselines.

Limitations. Output quality depends on the planner’s ability to decompose queries into effective tool calls with appropriate dependencies. A dual-model architecture (stronger reasoning model for planning, cheaper executor model for reflections) partially addresses this. The execution gate controls which tools execute and at what impact level, but does not control what information the LLM processes once tool results are in context—information flow control is orthogonal.

Future work

  • Procedural memory. Agent self-modification of its own prompts and planning strategies based on accumulated experience, following the procedural memory framework from cognitive architectures.
  • Semantic skill routing. Embedding-based tool selection to reduce planner context when the tool registry is large, presenting only semantically relevant tools for each query.
  • Cross-agent coordination. Extending the DAG model to distributed multi-agent execution via gossip-based peer-to-peer mesh networking, where each agent maintains its own execution gate and clearance authority.
